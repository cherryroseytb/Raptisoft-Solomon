# Project Ground Rules — Solomon's Boneyard Unity Copy

> 이 문서는 본 폴더(`Solomon's Boneyard/`) 작업의 **프로젝트 전용 그라운드룰**입니다.
> 글로벌 룰(`~/.claude/CLAUDE.md`)을 모두 따르되, 본 문서가 충돌 시 우선합니다.

---

## 1. 프로젝트 목적 (가장 중요)

### 1.1 최종 목표
**GitHub에 있는 원작(`Solomon's Boneyard`)을 Unity 엔진으로 재구현하여, 내가 직접 컨트롤·수정할 수 있는 카피 게임을 만드는 것.**

### 1.2 두 단계 진행
| 단계 | 목적 | 상태 |
|---|---|---|
| **Phase 1: 카피 (현재 단계)** | 원작과 외관·동작이 최대한 같은 Unity 클론 제작 | 진행 중 |
| **Phase 2: 변형 (이후 단계)** | 카피본 기반으로 독자 시스템·아트·룰 추가 | 대기 |

**현재는 Phase 1만 진행한다. Phase 2의 아이디어가 떠올라도 코드에 미리 반영하지 않는다.**

### 1.3 원작 정보
- **플랫폼:** 안드로이드 모바일이 원조. 최근 PC 버전 출시.
- **저장소 보관 빌드:** `SB.ver.1.2.2.build.1029` (안드로이드 APK, ARM v7a, hdpi)
- **원작 분석 자료:**
  - `SB_v1.2.2_Merged/` — APK 머지본 (리소스 확인용)
  - `cr-dev-log.md` — 원작 핵심 재미 분석
  - `development-log.md` — 카피 개발 히스토리
  - `TEST_CHECKLIST.md` — 사용자 직접 테스트 항목 누적

### 1.4 타겟 플랫폼
**안드로이드 우선.** 원작이 모바일이므로 입력·UI·해상도 모두 모바일 기준.
PC 빌드는 에디터 테스트 편의용 폴백만 유지.

## 2. 카피 충실도 원칙

### 2.1 관찰 기반 카피
원작의 소스 코드(C++ 네이티브)는 우리에게 없다. 따라서 다음 자료에서 동작을 역설계한다:
- APK에서 추출한 리소스(스프라이트, 아이콘, 사운드)
- 실제 플레이 영상·스크린샷
- 사용자의 플레이 경험에서 나온 메모

### 2.2 "잘 카피했다"의 정의
다음 항목이 원작과 **외관·체감상 일치**하면 카피 성공:
- [ ] 화면 레이아웃 (조이스틱·HUD·버튼 위치 비율)
- [ ] 캐릭터 이동·회전 속도감
- [ ] 마법 발사 주기·이펙트 색상·궤적
- [ ] 적의 등장 속도·체감 난이도 곡선
- [ ] 레벨업 시 선택지 카드 UI 구조
- [ ] 사운드 큐 타이밍

### 2.3 카피와 추측의 구분
원작에서 명확히 관찰되지 않은 동작은 **임의로 추가하지 않는다**. 필요 시 사용자에게 확인 요청 후 진행한다.
- 예: "원작에서 슬롯 전환이 자동인가, 버튼인가?" 같은 질문은 추측 대신 사용자에게 묻기.

### 2.4 저작권 / 에셋 정책
- 원작의 **스프라이트·사운드·아이콘을 그대로 복제·재배포하지 않는다.**
- APK에서 추출한 리소스는 **개인 학습·구조 파악용**으로만 참고. 최종 빌드에 포함 금지.
- 카피 코드에 원작의 클래스명·내부 변수명을 추측해 그대로 쓰지 않는다 (관찰 가능한 동작만 재현).

## 3. 작업 우선순위

다음 순서를 어기지 않는다:

1. **원작에 존재하는 기능을 카피** — 미구현 항목이 남아있다면 이게 최우선.
2. **카피된 기능의 정확도 개선** — 수치·이펙트·UX를 원작에 맞춤.
3. **카피 검증** — 사용자 테스트 체크리스트 통과.
4. **(승인 후) 변형 기능 추가** — Phase 2 시작 시점은 사용자가 명시.

## 4. 코드 작성 규칙 (본 프로젝트 한정)

### 4.1 폴더 구조
```
Solomon's Boneyard/
├── copygame/        Unity 스크립트 미러 (커밋 추적용)
├── SB.ver.*/        원작 APK (참고용, 수정 금지)
├── SB_v1.2.2_Merged/원작 머지본 (참고용, 수정 금지)
├── development-log.md  설계·구현 기록
├── TEST_CHECKLIST.md   사용자 직접 검증 항목 (누적)
└── CLAUDE.md           본 문서
```

### 4.2 Unity 프로젝트 위치
- 실제 프로젝트: `F:\Unity Projects\solomon-copy-game` (WSL에서는 `/mnt/f/Unity Projects/solomon-copy-game`)
- 코드 변경 시 `copygame/`에 먼저 작성 후 `Assets/Scripts/`로 복사. 양쪽 동기화 유지.

### 4.3 네임스페이스
모든 카피 코드는 `SolomonCopy.<카테고리>` 네임스페이스 사용. (예: `SolomonCopy.Player`, `SolomonCopy.Magic`)

### 4.4 데이터 중심 설계
가능하면 **ScriptableObject**로 데이터를 코드 밖으로 빼낸다. 마법·적·업그레이드 등 원작에서 다양한 인스턴스가 등장하는 요소는 모두 SO 자산화.

### 4.5 한글 주석
스크립트 상단에 **한글로 Unity 에디터 셋업 가이드**를 적는다 (필수 컴포넌트, 인스펙터 연결, SO 생성 메뉴 경로). 사용자가 Unity 작업할 때 참고용.

### 4.6 모바일 우선 입력
모든 입력은 **터치 기반**이 기본. 키보드·마우스는 `#if UNITY_EDITOR || UNITY_STANDALONE` 폴백으로만 둔다.

## 5. 커뮤니케이션 룰

### 5.1 카피 항목 vs 신규 항목 명시
새 기능을 제안할 때는 항상 명시한다:
- **[CP]** = 원작에 있는 기능을 카피
- **[+]** = 원작에 없거나 확신할 수 없는 신규 추가

### 5.2 원작 확인이 필요한 경우
"원작에서 이 부분이 어떻게 동작하는지 확인 부탁드립니다" 형식으로 사용자에게 질문. 추측 진행 금지.

### 5.3 테스트 체크리스트
새 기능 구현 시마다 `TEST_CHECKLIST.md`에 사용자 검증 항목을 **누적 추가** (덮어쓰지 않음).

### 5.4 개발 히스토리
의미 있는 마일스톤마다 `development-log.md`에 날짜·결정사항·구현 범위 기록.

## 6. 알려진 한계 / 주의

- 원작 소스 없음 → 정확한 수치·확률·내부 로직은 관찰 추정.
- APK 디컴파일은 학습 범위에서만 수행. 추출 리소스 재배포 금지.
- 현재 Unity 프로젝트(`F:\...`)는 사용자 로컬 윈도우 드라이브. 빌드·테스트는 사용자 환경에서 수행.
